Problem TGL : 2026-07-10
Subject : DailyOnesheet.jsx (Line 1, Line 2, Line 3, Line 4)
Deskripsi troubel : Hasil analisa bug pada seluruh DailyOnesheet.jsx semua line

--- BUG-DOS-01 | SEMUA LINE (L1, L2, L3, L4) ---
Kategori : KRITIKAL - DATA SALAH / SALAH TABEL
Lokasi   : executeSearch() -> fetchOnesheetData(inputDate, user)
Masalah  : fetchOnesheetData() meneruskan objek `user` langsung dari useAuth() ke
           backend (api/fetch-data.js). Backend menentukan tabel yang diquery berdasarkan
           user.line (baris 12: const rawLine = user?.line || user?.plant || "4").
           Jika user login dengan line yang berbeda (misal user Line 4 membuka
           DailyOnesheet Line 1), backend akan tetap query ke tabel line 4 (oee_line4_zonec,
           downtime_line4_zonec, dst). Data yang ditampilkan di DailyOnesheet Line 1
           adalah data dari line yang salah.
Dampak   : Semua line (L1, L2, L3, L4) akan menampilkan data dari tabel yang
           bergantung pada sesi login user, bukan tabel milik line yang sedang dibuka.

--- BUG-DOS-02 | LINE 1, LINE 2, LINE 4 ---
Kategori : SEDANG - DELAY DOWNLOAD TERLALU LAMA
Lokasi   : handleDownloadJPG() -> setTimeout(..., 10000)
Masalah  : DailyOnesheet Line 1 (L423), Line 2 (L423), Line 4 (L422) menggunakan
           setTimeout dengan delay 10.000 ms (10 detik) sebelum proses unduh PNG dimulai.
           User harus menunggu 10 detik minimal setiap kali menekan tombol UNDUH.
Catatan  : Line 3 sudah benar dengan delay hanya 800 ms (L428).

--- BUG-DOS-03 | LINE 1 ---
Kategori : SALAH DATA - Label Line di Header Report Salah
Lokasi   : DailyOnesheet.jsx Line 1 -> JSX line 455
Kode     : {user?.line || 2}
Masalah  : Fallback default Line tampil sebagai "2" padahal ini adalah DailyOnesheet
           Line 1. Seharusnya fallback adalah "1".
Dampak   : Header laporan cetak/PNG menampilkan "Line : 2" jika user belum login atau
           user.line tidak terdefinisi.

--- BUG-DOS-04 | LINE 4 ---
Kategori : SALAH DATA - Label Line di Header Report Salah
Lokasi   : DailyOnesheet.jsx Line 4 -> JSX line 454
Kode     : {user?.line || 2}
Masalah  : Fallback default Line tampil sebagai "2" padahal ini adalah DailyOnesheet
           Line 4. Seharusnya fallback adalah "4".
Dampak   : Header laporan cetak/PNG menampilkan "Line : 2" jika user.line tidak terdefinisi.

--- BUG-DOS-05 | SEMUA LINE ---
Kategori : SEDANG - REACT HOOKS WARNING (Exhaustive-deps)
Lokasi   : useEffect(() => { if (user) executeSearch(); }, [])
Masalah  : useEffect dipanggil dengan array dependency kosong [] namun menggunakan
           `user` dan `executeSearch` (yang bergantung pada `user`, `inputDate`).
           Jika user belum siap saat mount pertama (async auth check), executeSearch()
           akan dipanggil dengan user = null -> fetchOnesheetData(inputDate, null) ->
           backend menggunakan default lineNum "4" -> data salah.
           Selain itu, jika user berganti (logout/login ulang), data tidak di-refresh.

--- BUG-DOS-06 | LINE 3 ---
Kategori : RINGAN - Field mapping Volume berbeda dengan line lain
Lokasi   : useZoneCProcessor & useZoneFProcessor -> filteredReject filter
Kode     : r.vol_botol || r.volume_botol  (Line 3)
           r.volume_botol                 (Line 1, 2, 4)
Masalah  : Line 3 menggunakan field fallback ganda `r.vol_botol || r.volume_botol`
           karena struktur kolom DB Line 3 berbeda (`vol_botol` bukan `volume_botol`).
           Ini sudah ditangani di Line 3, namun jika nama kolom berubah di DB,
           filter akan gagal tanpa pesan error yang jelas.

--- BUG-DOS-07 | LINE 3 - Zone F ---
Kategori : SEDANG - Field Data Tidak Konsisten
Lokasi   : useZoneFProcessor -> filteredReject.forEach -> data[g].pot
Kode     : data[g].pot += (parseFloat(r.at_sub || r.av_sub) || 0) / 60
           data[g].out_counter += parseFloat(r.cnt_sub || r.vi_sub || r.input_botol_chamber) || 0
Masalah  : Line 3 Zone F menggunakan 3 lapisan fallback field yang berbeda-beda
           (r.at_sub || r.av_sub, r.cnt_sub || r.vi_sub || r.input_botol_chamber).
           Ini menunjukkan ketidakpastian nama kolom di tabel oee_line3_zonef.
           Jika semua field bernilai 0, POT dan Output akan terhitung 0, menyebabkan
           seluruh perhitungan OEE Zone F di Line 3 menjadi 0%.

--- BUG-DOS-08 | LINE 2 ---
Kategori : SEDANG - Volume default tidak sesuai Line 2
Lokasi   : DailyOnesheet.jsx Line 2 -> useState("500 ML")
Masalah  : Line 2 memiliki VOLUMES = ["25 ML", "100 ML", "250 ML", "500 ML", "1000 ML"].
           Default volume yang dipilih saat halaman dibuka adalah "500 ML".
           Jika produksi Line 2 dominan menggunakan "25 ML" atau volume lain,
           data awal yang ditampilkan akan selalu kosong (filter volume tidak cocok)
           hingga user secara manual mengubah pilihan volume.

Deskripsi Solusi :
 - BUG-DOS-01 : Tambahkan objek eksplisit lineNUser { ...user, line: "X" } di setiap
   DailyOnesheet sebelum memanggil fetchOnesheetData (sama seperti fix di INPUTC/INPUTF).
 - BUG-DOS-02 : Ubah setTimeout delay dari 10000 ms menjadi 800 ms di Line 1, Line 2, Line 4.
 - BUG-DOS-03 : Ubah fallback dari `|| 2` menjadi `|| 1` di DailyOnesheet Line 1.
 - BUG-DOS-04 : Ubah fallback dari `|| 2` menjadi `|| 4` di DailyOnesheet Line 4.
 - BUG-DOS-05 : Tambahkan user ke dalam dependency array useEffect atau gunakan
   useCallback untuk executeSearch agar refresh otomatis saat user siap.
 - BUG-DOS-06 : Dokumentasikan nama kolom DB per line, tidak perlu perbaikan kode
   jika sudah berjalan.
 - BUG-DOS-07 : Verifikasi nama kolom di oee_line3_zonef dan standarisasi.
 - BUG-DOS-08 : Ubah default volume Line 2 sesuai produk yang paling sering dipakai.
Resiko Solusi : Rendah untuk BUG-DOS-02/03/04/08. Sedang untuk BUG-DOS-01/05
               (perlu verifikasi user object di semua line sebelum deploy).
Status : BUG-DOS-01 SOLVED (line1User, line2User, line3User, line4User diterapkan dan terverifikasi build sukses). Bug lainnya masih IDENTIFIED.
